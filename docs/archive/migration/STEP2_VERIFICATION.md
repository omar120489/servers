## üîß Technical Implementation Details

### **P&L Calculation:**
```typescript
netProfit = grossRevenue - directCost
```

### **Status Values Supported:**
- Frontend sends: `"Closed Won"`
- Backend accepts: `"Closed Won"` or `"Won"`
- Validation triggers on transition FROM any non-won status TO won status

### **Data Flow:**
1. User clicks "Mark Won" ‚Üí Modal opens
2. User enters financials ‚Üí Net profit calculates in real-time
3. User clicks "Confirm" ‚Üí Frontend calls `updateDeal(id, { status, grossRevenue, directCost })`
4. Service maps to snake_case ‚Üí `{ status, gross_revenue, direct_cost }`
5. Backend validates ‚Üí If transitioning to won, checks financials
6. Backend computes `net_profit` ‚Üí Stores all three values
7. Backend returns updated deal with `net_profit`
8. Service maps to camelCase ‚Üí `{ status, grossRevenue, directCost, netProfit }`
9. UI updates to show new status and financial details

---

## üö® Edge Cases Handled

‚úÖ **Zero values:** Rejected (both gross revenue and direct cost must be > 0)  
‚úÖ **Negative values:** Rejected for inputs, but negative net profit is valid  
‚úÖ **Empty fields:** Confirm button disabled until both fields filled  
‚úÖ **Non-numeric input:** HTML5 type="number" prevents non-numeric entry  
‚úÖ **Already closed deals:** "Mark Won" button hidden if status is already "Closed Won" or "Won"  
‚úÖ **Deal without ID:** Button and modal handlers check for deal.id before proceeding  
‚úÖ **API errors:** Frontend catches errors and logs to console (toast integration ready)
